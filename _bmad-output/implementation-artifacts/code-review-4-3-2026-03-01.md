## 🟡 HIGH ISSUES

### 1. Timeline can drop actor identity for audit events that store actor in `actorId` column

**Finding:** Timeline actor extraction only reads actor fields from payload (`actor`, `actor_id`, `user_id`). Governance audit events created by approval/send flows set actor in the audit event `actorId` field, not payload, so actor can be rendered as unknown even when it exists in storage.

**Evidence:**
- `src/server/services/approval-service.ts:58-62` and `src/server/services/approval-service.ts:92-96` pass `actorId` to audit creation.
- `src/server/services/lead-service.ts:86-94` maps governance events to timeline items but does not include `actorId`.
- `src/app/api/leads/[id]/timeline/route.ts:48-53` derives actor only from payload fields.

**Impact:** Violates the story goal of auditable, clear history (actor attribution is a core governance field).

**Recommendation:** Include `actorId` in timeline service mapping and resolve actor in route as `event.actorId ?? payload.actor*`.

---

### 2. `action.sent` transition text is hardcoded to `approved -> sent`, which is false for low-priority direct sends

**Finding:** Timeline transition mapping always returns `Draft: approved -> sent` for `action.sent`, but low-priority leads can send directly without prior approval.

**Evidence:**
- `src/app/api/leads/[id]/timeline/route.ts:42-43` hardcodes transition.
- `src/server/services/approval-service.ts:107-128` allows low-priority direct send and still emits `action.sent`.

**Impact:** Audit trail can present an incorrect state transition, reducing trust in governance records.

**Recommendation:** Derive transition from payload/context (or omit when unknown) instead of hardcoding a single path.

---

## 🟠 MEDIUM ISSUES

### 3. Story says ad hoc history was replaced, but `Override trace` block remains alongside `DecisionTimeline`

**Finding:** Lead detail still renders a separate override-history section after integrating `DecisionTimeline`.

**Evidence:** `src/app/(dashboard)/lead/[id]/page.tsx:627-648`.

**Impact:** Duplicate governance history surfaces can diverge and weaken the intended “single auditable timeline” UX.

**Recommendation:** Remove or clearly scope the legacy override block once timeline coverage is confirmed complete.

---

### 4. Task marked complete for keyboard operability assertions is only partially validated in tests

**Finding:** Story marks keyboard operability assertions complete, but tests validate click + `aria-expanded`; they do not assert keyboard interaction behavior for entry expansion.

**Evidence:** `src/features/governance/components/DecisionTimeline.test.tsx:56-70` uses click-based expansion only.

**Impact:** Accessibility regressions in keyboard activation could pass unnoticed.

**Recommendation:** Add a test using keyboard events (`Enter`/`Space`) on expandable controls and assert expanded state transitions.
