## 🟠 MEDIUM ISSUES

### 3. Error envelope exposes raw error messages to client (information disclosure)

**Finding:** When the KPI API fails, the catch block passes `err.message` into `createErrorResponse`'s `details` array. This sends internal error text (e.g., "Connection refused", Prisma stack traces) to the client, which can leak implementation details.

**Evidence:** `src/app/api/kpi/route.ts:37` — `[err instanceof Error ? err.message : String(err)]` is sent to the client in the JSON response.

**Recommendation:** Log the full error server-side for debugging; for the API response, use an opaque identifier or generic message in `details`. Architecture says "Log technical details server-side; show concise user-facing message." The `message` is correct; `details` should avoid raw `err.message` in production.

*Note: This pattern exists across multiple API routes (sla, leads, risk-pulses, etc.); fixing here would improve KPI route; broader fix is a separate effort.*

**Evidence:** `KPISummaryPanel.tsx:42-44` — single "Loading…" text node; `insights/page.tsx` passes `loading={true}` to panel.

**Recommendation:** Add a simple skeleton grid (4 placeholder cards with consistent dimensions) during loading to prevent layout jump and align with UX loading patterns.

### 6. NFR4 (3s p95) has no automated validation

**Finding:** Story NFR4: "KPI panel loads within 3 seconds for 95% of requests." There is no performance test or assertion. The aggregation is lightweight (single `Promise.all`), but regressions (e.g., N+1, extra queries) would go undetected.

**Recommendation:** Add a lightweight perf assertion in integration tests (e.g., KPI response time < 3s under normal load) or document as manually validated. Optional for MVP.

### 7. Insights page uses inline styles instead of design tokens

**Finding:** `insights/page.tsx` and `KPISummaryPanel.tsx` use inline `style={{}}` and CSS variables (`var(--foreground)`). Architecture and UX point to `src/styles/tokens.ts` and theme. Triage and other pages may use a more consistent pattern.

**Evidence:** `insights/page.tsx:33-39`, `KPISummaryPanel.tsx` — heavy inline styling.

**Recommendation:** Align with project styling patterns (tokens, MUI, or shared components) for consistency. Low priority if current approach is acceptable for hackathon scope.

---

### 8. KPI API: tenant_id=empty string edge case

**Finding:** If the client sends `?tenant_id=` (empty string), `searchParams.get("tenant_id")` returns `""`. The code uses `tenantIdParam ?? getOrCreateDefaultTenant()`, so `""` is used (falsy but not null/undefined). Then `isValidUuid("")` is false, and we return 400. Correct behavior, but worth noting that `""` is treated as "provided but invalid" rather than "use default."

**Recommendation:** Consider explicitly treating `""` as "use default" (e.g., `tenantIdParam && tenantIdParam.trim() ? tenantIdParam : getOrCreateDefaultTenant()`). Optional refinement.

---

### 9. KPISummaryPanel: error state uses "!" as icon (weak NFR13)

**Finding:** NFR13: "Metrics use icon + text; never color-only." The unavailable state uses `<span aria-hidden="true">!</span>` as the icon. A more recognizable icon (e.g., ⚠ or an SVG) would improve accessibility and consistency with other components (e.g., SLASafetyIndicator).

**Recommendation:** Use the same warning/error icon as SLASafetyIndicator or another established pattern. Low impact.
