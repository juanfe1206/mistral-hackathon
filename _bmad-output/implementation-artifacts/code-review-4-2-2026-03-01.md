## 🟡 HIGH ISSUES

### 1. At-risk recovery UI disappears when lifecycle is `at_risk` but no active pulse is returned

**Finding:** The recovery controls are now gated by both `lead.lifecycle_state === "at_risk"` and `activePulse`. If backend data is temporarily inconsistent (lead marked at-risk but pulse list empty), the page shows no recovery composer and no lifecycle actions.

**Evidence:** `src/app/(dashboard)/lead/[id]/page.tsx:505` gates rendering with `&& activePulse`.

**Impact:** Operator cannot generate/approve/send recovery or mark recovered/lost for leads still marked `at_risk`, creating an operational dead-end.

**Recommendation:** Render `AtRiskPulseBanner` + `ConciergeReplyComposer` whenever lifecycle is `at_risk`; when pulse is missing, use a safe fallback reason/timestamp and keep actions enabled.

---

## 🟠 MEDIUM ISSUES

### 2. Tone selector has no effect on generated draft

**Finding:** UI exposes tone selection (`warm/neutral/direct`), but generate requests ignore selected tone and backend contracts do not accept it.

**Evidence:** `src/app/(dashboard)/lead/[id]/page.tsx:170-174` sends only `{ lead_id }`.

**Impact:** UX communicates controllable tone, but output cannot change based on that control. This is misleading and can reduce operator trust.

**Recommendation:** Include `tone` in generate request schema and thread it through `POST /api/replies/generate` into `reply-service` prompt construction; or hide/disable tone until supported.

---

### 3. “Integration test” does not cover page/API integration claimed in story

**Finding:** The new integration test validates only a local state harness around `ConciergeReplyComposer`; it does not exercise `lead/[id]/page.tsx`, route handlers, or real fetch transitions.

**Evidence:** `tests/integration/lead-detail-recovery-flow.test.tsx:15-58` defines a local `RecoveryHarness` and never renders the lead detail page.

**Impact:** Regressions in page wiring (API payloads, error handling, lifecycle gating) can ship undetected despite a green “integration” suite.

**Recommendation:** Add at least one integration test rendering `src/app/(dashboard)/lead/[id]/page.tsx` with mocked fetch responses to verify generate -> approve -> send and low-priority direct-send branches through the actual page logic.
