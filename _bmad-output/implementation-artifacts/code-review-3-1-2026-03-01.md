## 🟡 HIGH ISSUES

### 1. Lead detail route: SLA error handling not tested [leads/[id]/route.ts:69–77]

**Finding:** When `getLeadSlaStatus` throws (DB error, timeout), the route catches and returns `sla_status: null`. The UI correctly shows "SLA - Retry" via `unavailable={!lead.sla_status}`. **However, there is no test that verifies this behavior.** AC2 requires fallback when SLA is unavailable; without a test, regressions could slip in.

**Evidence:** `tests/api/leads-detail.test.ts` has no case where `mockGetLeadSlaStatus.mockRejectedValue(...)`.

**Recommendation:** Add a test: when `getLeadSlaStatus` rejects, route returns 200 with `sla_status: null` and lead data otherwise complete.

---

### 2. Queue summary vs leads list: different limits (500 vs 100) [sla-service.ts:94, 140]

**Finding:** `getQueueSlaSummary` uses `limit: 500` while `getLeadsWithSlaStatus` uses `limit: 100`. The triage page shows leads from `/api/leads` (100) and SLA header from `/api/sla` (500). Header counts ("2 at risk", "SLA Safe 92%") can refer to leads not visible on screen, causing confusion.

**Evidence:**
- `getQueueSlaSummary`: `findLeadsByTenant(tenantId, { limit: 500 })`
- `getLeadsWithSlaStatus`: `findLeadsByTenant(tenantId, options ?? { limit: 100 })`

**Recommendation:** Either use the same limit for both, or pass `limit` from leads API to SLA summary so they share the same scope. Document the chosen behavior.

## 🟠 MEDIUM ISSUES

### 3. UX spec: “Clicking opens filtered list” not implemented [SLASafetyIndicator.tsx]

**Finding:** UX spec (ux-design-specification.md §SLASafetyIndicator) states: “Interaction Behavior: Clicking opens filtered list of affected leads.” Neither `QueueSlaIndicator` nor `LeadSlaIndicator` implements click-to-filter. Users cannot drill into at-risk leads from the queue SLA badge.

**Recommendation:** Add `onClick` to `QueueSlaIndicator` when `atRisk > 0` and wire it to a filter/navigation (e.g. triage view filtered by breach/warning/breach-risk). If out of scope for MVP, document as deferred.

---

### 4. Lead detail: unclear “unavailable” vs “N/A” in some flows [lead/[id]/page.tsx:340–345]

**Finding:** Low-priority leads get `sla_status: { status: "n_a", ... }`; API errors yield `sla_status: null`. UI uses `unavailable={!lead.sla_status}` so both non-null and null are handled. For `null` the message is "SLA - Retry", which is correct. The distinction could be clearer if the API returned an explicit `sla_unavailable: true` flag when SLA failed, instead of overloading `null`.

**Recommendation (optional):** Consider `sla_status: null | T` with `sla_fetch_failed?: boolean` in the response when SLA lookup fails, so the frontend can show a more specific message. Current behavior is acceptable; this is an improvement suggestion.

---

### 5. GET /api/leads response shape changed [api/leads/route.ts]

**Finding:** The story says “extend response to include sla_status” and “add as optional enrichment.” The implementation fully replaces the previous lead-fetching logic with `slaService.getLeadsWithSlaStatus`, returning a new mapped shape. The triage page expects this shape and it works, but any other consumers of `/api/leads` might expect the old schema (e.g. `classifications` nested). No other consumers were found in the repo.

**Recommendation:** Add an API changelog or note that `/api/leads` response shape was updated for 3.1. If there are external clients, ensure backward compatibility or versioning.

---

### 6. No UI test for SLA fallback (Task 9) [tests/]

**Finding:** Task 9 requires “Test UI fallback when SLA unavailable.” There is no component, integration, or e2e test that mocks SLA failure and asserts “SLA temporarily unavailable” + retry behavior. The API test (`sla.test.ts`) covers the error envelope; the triage page logic is manually verified only.

**Recommendation:** Add a component test for `QueueSlaIndicator` and `LeadSlaIndicator` with `unavailable={true}` to confirm the fallback UI renders correctly. Optionally add an integration test that mocks `/api/sla` failure and checks the triage header.

## 🟢 LOW ISSUES

### 7. Icons are ASCII characters, not icon components [SLASafetyIndicator.tsx:36–49]

**Finding:** NFR13 requires “icon + text + color.” Implementation uses `"+"`, `"!"`, `"X"`, `"~"`, `"-"` as `aria-hidden` content. These work for screen readers via `aria-label` but are not semantic icon components (e.g. Lucide/MUI icons).

**Recommendation:** Replace with proper icon components for consistency and future theming. Low priority for MVP.

---

### 8. “Responded in time” vs “Responded in Xm” [SLASafetyIndicator.tsx:55]

**Finding:** Story example: “Responded in 2m” (safe). Implementation uses “Responded in time” when `first_response_at` exists. Showing actual minutes (e.g. “Responded in 2m”) would align better with the UX spec and be more informative.

**Recommendation:** Compute minutes from `created_at` to `first_response_at` and display “Responded in Xm” when safe.

---

### 9. UX spec: trend arrow missing [SLASafetyIndicator.tsx]

**Finding:** UX spec anatomy: “Status chip, trend arrow, time-to-breach metric.” No trend arrow is implemented. Only status and time-to-breach are shown.

**Recommendation:** Add an optional trend indicator (e.g. improving vs worsening) if/when historical SLA data is available. Defer if not MVP-relevant.

---

### 10. SLA route: tenant_id from query vs auth [api/sla/route.ts]

**Finding:** Tenant is resolved via `tenant_id` query param or `getOrCreateDefaultTenant()`. Architecture expects tenant “from auth/session” for protected endpoints. Current behavior matches other routes (e.g. leads) and is consistent, but auth-based tenant resolution is not used.

**Recommendation:** Confirm with architecture/security: if endpoints should use auth context, add a follow-up story. Otherwise document that query param + default tenant is acceptable for current auth setup.

---

### 11. Minor: redundant mock in leads-detail.test.ts [leads-detail.test.ts:31]

**Finding:** `mockGetLeadSlaStatus` is set in `beforeEach` but not all tests assert on SLA. This is fine; it prevents unhandled rejections. No change needed; just noted.
